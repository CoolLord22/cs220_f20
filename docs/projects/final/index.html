Final Project

caution
  • You are expected to work in a team of three.
  • Release date: Wednesday Nov 18th
  • Project Due: Wednesday Dec 9th 11pm EST
  • Individual Report Due: Thursday Dec 10th 11pm EST
Learning Objectives

This assignment will give you practice with writing C++ classes using:

  • inheritance,
  • dynamic binding,
  • pure virtual functions,
  • and abstract base classes
Collaboration

This project is to be completed in pre-registered teams of three students. No collaboration is permitted outside a team. It is your responsibility to work effectively as a team. Each student will be required to submit a feedback file once the project has ended detailing what contributions each student made to the project (see Contributions Submission section below).

Starter Code

This homework comes with starter source files found in the public repository under the folder named final_project/. Teams should ensure they have a proper copy of these files before they start writing any code.

Problem Description#

For this final project you will work in teams to write a program for the card game Skip-Bo. To get started, see this video for a quick tutorial on the game and this site for further helpful explanation. There are also videos other explaining the game, and mobile versions (can be found on Play Store or App Store) you can try out for yourself.

There are several variations of the game. You will be implementing a multi-player version, for between 2-6 players (no partner play). You also will not implement scoring - just one game played through from start to a winner. There are specific requirements (below) regarding game initialization, saving and loading that you must follow so that we can do some automated testing on your project. Make sure you read all the details below and use the provided code!

Object-Oriented Design#

The diagram below gives a suggested starting point for the inheritance-related classes you should write to represent the various types of Card collections for the game. PlayPile can be used to represent stock piles and also discard piles. You’ll also need to figure out where a Hand class fits in.

Think about how each of these classes is used in the game, adding necessary functions to support their uses (and any data), but only in the classes that absolutely need them. For example, only the DrawPile gets shuffled. Also remember to differentiate how they each get displayed. Take advantage of inheritance as much as possible without abusing it.

Skeleton Code#

We are providing you with the following starter files (Items 1-4):

  1. Card.h/Card.cpp - contains Card class, which has:

    • private int value
    • constructor Card(int v)
    • int getValue() const
    • std::string toString() const // for output to file
    • void display() const // how to see in game play

  1. Pile.h/Pile.cpp - contains:

    • private vector<Card> pile;
    • public virtual addCard(const Card& c); // puts c at "top", regardless of Pile type
    • public toString() const; // outputs entire contents of Pile, needed for state saving. You must use this in order to create consistent saved states that we can use for testing and autograding.
    • You'll need to write void readIn(istream &), to perform the inverse of toString in a sense -- make the Pile's contents match the state given by a particular stream.
    • public virtual void display() = 0; // outputs the view of this Pile to cout as appropriate for Pile subtype during game play, but this will need to be implemented in derived classes. We’ve provided the necessary pieces of those classes also, to insure that your interface is exactly the same as ours.
      • DrawPile::display() will only show size of Pile, no actual Card values
      • FaceUpPile::display() will display the top card value using Card::display and the pile size
      • Hand::display() will show all cards and "--" for missing cards

  2. Player.cpp - We have provided toString and display operations for consistency with gameplay and state saving. Do not modify these functions as we will use them for autograding. You’ll need to figure out what to put in Player.h in order to support this code, as well as everything else the class needs to handle. You'll need to write void readIn(istream &), to perform the inverse of toString in a sense -- make the Pile's contents match the state given by a particular stream.

  3. SkipBoGame.cpp - We have provided toString and display operations for consistency with gameplay and state saving. Do not modify these functions as we will use them for autograding. You’ll need to figure out what to put in SkipBoGame.h in order to support this code, as well as everything else the class needs to handle. You'll need to write void readIn(istream &), to perform the inverse of toString in a sense -- make the Pile's contents match the state given by a particular stream.

  4. main.cpp - You will write this one. This should be your main driver that collects one of 2 different sets of command line args depending on whether a new game is being started, or a saved game is being reloaded. In both cases, the first argument will be “true” or “false” indicating whether the draw pile should be shuffled (true, for regular (random) game play) or not (false, for testing purposes).

    • New game args will be shuffle, numPlayers, stock pile size, and deck start file. These values should be used to construct a new SkipBoGame object. Players should always be named Player0, Player1, etc. The deck start file is a plain text file with an ordering of all the cards in the deck. SkipBo wildcards will be represented with a 0 in these files. Our provided standard deck0.txt file should be used as a default. Once the game is initialized, play should begin and continue until it is interrupted and saved, or there is a winner.
    • Reload game args will only be shuffle and savedGame.txt file which contains a complete saved game state. Once loaded, play should continue until it is interrupted and saved, or there is a winner.

    See sampleMainErrors.txt below (in Sample Runs) for a full list of main command-line error conditions that must be handled, and how. See sampleLoad.txt below (in Sample Runs) for a demonstration of loading a saved game state.

  5. Game play - user interface: (see Sample Runs section below)

    • First Player in a new game: In shuffle==true mode, the first player should be chosen randomly. In test mode, Player0 should always have the first turn. In a reload situation, the savedGame file will determine whose turn it is.

    • Game display: display the game to start, then refresh the display after every player move and when continuing to the next player’s turn.

    • Player turns: turn will always pass to the next player (+1) in a round-robin fashion. At the start of every turn, cards from the top of the draw pile will automatically be added to a Player’s hand as needed so that they have 5. When the game begins and at the end of every turn (when the current player moves a card from their hand to a discard pile), check to see if the players will continue to (p)lay, (s)ave the game, or simply (q)uit playing this game without saving. When saving game state: player will need to input the file name for the saved game.

    • While a Player is taking their turn which may include multiple moves, two options must be available: (m)ove a card from one pile to another or (d)raw enough cards to fill hand. The draw option is only legal if the hand is empty. After each move check to see if that player has won (stock pile is empty). If so, the game should end and the winner be announced by name.

    • Move input: Player cards/piles will be numbered according to the scheme below and the 4 shared build piles will be indicated by letters a-d. Players will indicate which card to move and where to put it by these numbers and letters:

      • 0 = stock pile (can only go to a build pile)
      • 1-4 = discard piles (from discard to build or from hand to discard)
      • 5-9 = cards in hand (to build or to discard)
      • a-d = shared build piles, in left to right order

    A sample move would be m 0 c to move from the player’s stock pile to build pile c. Another example is m 6 3 to move a card from the hand to a discard pile, ending the player’s turn. You must check that each requested move is legal and reject it otherwise with the error message illegal command, try again. There should always be at least one space between m and the "move-from" location, and at least one space between the "move-from" and "move-to" locations in the move command. If two moves are typed in at one prompt, then two moves should be made, but the "in-between" prompt will simply appear after the second move is made.

    • When the draw pile size gets down to zero, we'll use all build piles which are complete at that moment to replenish it. If shuffle is true, then the cards are shuffled when used to replenish the draw pile. Otherwise, we'll stack the build piles together as-is (no shuffling or interspersing cards from different build piles together), so that the top card of the draw pile is a 12, the 2nd card is a 11, etc., and the 12th is a 1. If two build piles were complete, then the pile would have 24 cards, and the 13th card is a 12, the bottom (24th) card is a 1.
    • After a player wins, the game just exits without further prompting.

Requirements, Building, and Submissions#

Further Requirements and Assumptions
  • You can manipulate the starter code however you wish. Just make sure your code works using our user interface.
  • There is always a supplied deck. Your program should always get its deck contents from the supplied command-line filename. Then you can look at the shuffle argument's value to determine whether to randomize the order of the cards in that file.
  • You can assume that all deck text files contain valid decks.
  • When a build pile is finished, add it to the top of the other set-aside (completed) build piles. This is the most natural in terms of mimicking physical game play. If shuffle state is false, they never get shuffled. If it is true, they get shuffled when added to the draw pile.
  • When saving game state, add the completed build piles to the BOTTOM of the draw pile before saving game state. If shuffle mode is true, it should also then shuffle the draw pile before savnig. If shuffle mode is false, then the current draw pile will then be played out before the finished build piles get used, preserving the order as if the game were not interrupted by saving.

Specific Requirements#

  • Use header guards in all header (.h) files.
  • Do not use using in header (.h) files.
  • In C++ source files (.cpp files), you may import individual symbols using statements like "using std::string".
  • Do not use "using namespace <id>", either in headers or in source files.
  • All variables must be declared inside functions. No variables should be global or extern.
  • You may not use auto.
  • You may use the C++ library code (e.g., http://www.cplusplus.com/reference/algorithm/random_shuffle/) if you wish, or you may write your own shuffling function.

Makefile & Compiling#

You must create and submit a Makefile with a rule that creates a target executable called skipbo. The target should compile with no errors or warnings using the typical C++ compilation command: g++ <source> -Wall -Wextra -std=c++11 -pedantic. Do not use any special libraries that require different additional/different compiler options.

The autograder does an automatic check to see if it can compile the target. It is very important that you make sure your submission successfully compiles, otherwise your final score for the entire assignment will be 0.

Git log#

In the team repository, use git add, git commit and git push regularly, with meaningful commit messages. All team members should be writing code, so all team members should have a record of commits in the repository.

Code Submission (Team)#

Submissions of your project should be made on Gradescope. This will be a group submission, where each team member's name is indicated to Gradescope as part of the submission.

Your submission must include a copy of the output of git log showing at least six commits per team member to the repository. Save the git log output into a file called gitlog.txt (e.g. by doing git log > gitlog.txt).

Your submission must also include your Makefile as described above, and a README file explaining anything the graders should know about your submission, including any missing/incomplete functionality.

tip

Note that while we do absolutely expect you to write code that tests your implementation, you are not expected to submit your tests to us for grading.

Contributions Submission (Individual)#

Separately, after your team's code is submission is complete, each team member should individually create a brief text file called contributions.txt and submit on Gradescope via the corresponding link.

Individual Report Structure

The individual report file should have up to three sections:

  • A section titled TEAM which lists each team participant’s name and JHED id
  • A section titled CONTRIBUTIONS (required): A brief summary of which team members contributed what to the code submission.
  • A section titled ABOVE AND BEYOND (optional): If you feel that a team member contributed a particularly large amount of effort to the final product, name them here. You may name at most one team member. Do not name yourself.
info

Note that the deadline for this individual submission is 24 hours after the code submission deadline. Each team member must complete this individually and submit their own version of this file (in their own words) in order to get credit for their work on the final project.
