Homework 7

caution
  • You are expected to work individually.
  • Due: Friday November 20th at 11pm EST (Baltimore time).
  • This assignment is worth 60 points.
Learning Objectives

To practice with:

  • C++ operator overloading
  • C++ templates,
  • C++ exceptions,
  • C++ dynamic memory allocation
info

This homework comes with three starter source filels (CQueue.h, CQueueTest.cpp, and TQueueTest.cpp). You should get these files from the public repository under homework/hw7. Make sure you have a copy of these files before you start writing any code.

Part 1: A Character Queue#

Queue is a simple data structure that stores arbitrary objects and restricts their access to "first in, first out" (FIFO). You can read more about queues here. We can implemente a queue using a linked-list that we have learned in class. To start this homework, check the comments and class definition of CQueue in CQueue.h, which is a dynamic queue of characters. Your task is to implement this class.

Below are a list of functions you need to implement and some tips for the implementation:

  1. Default constructor: this function should create an empty queue, i.e. _head should be nullptr.

  2. Non-default constructor: this function takes a character input and create a queue with one element. (Hint: to create a new _Node, you may use list-initializiation, e.g. _Node({'a', nullptr}).)

  3. Copy constructor: applying the rule of 3, we have a non-trival destructor, we should implement the copy constructor for a deep copy.

  4. operator=: applying the rule of 3, we have a non-trival destructor, we should implement the assignment operator for a deep copy.

  5. Destructor: this function should deallocate all the allocated memory.

  6. Enqueue: this function takes a character and add it to the queue. i.e. a new _Node is allocated.

  7. Dequeue: this function returns and removes the first element in the queue. i.e. a _Node is deallocated. This function should also throw an exception if the queue is empty (i.e. nothing to dequeue).

  8. operator<<: this overloaded operator should do the same as Enqueue. (Hint: you may reuse Enqueue.)

  9. operator>>: this overloaded operator should do the same as Dequeue. (Hint: you may reuse Dequeue.)

  10. operator==: this overloaded operator compares two queues and returns true if the queues are the same node by node; otherwise return false.

  11. operator+: this overloaded operator concatenates two queues.

  12. std::stream& operator<<: this overloaded operator prints a queue in the this format: <data1> -> <data2> -> ... -> <datax> -> NULL\n. For example, if we have queue having three elements a, b, and c. This function should print this string: a -> b-> c -> NULL\n.

You are given a file named CQueueTest.cpp that provide basic tests for your implementation. Once you are done implmenting CQueue.cpp, you should compile and run cQueueTest to make sure that all test cases pass. Keep in mind that these are minimal cases. You should develop your own test cases and test your program thoroughly.

Part 2: A Templated Priority-or-not Queue#

For the second and last part of this homework, you will generalize the CQueue from part 1 to create a templated priority-or-not queue. Note that you will write your own header file for this part. Your queue class must be called Queue and be declared in Queue.h and fully defined (i.e. implementation of functions longer than one line should occur) in a file named Queue.inc. Note that the file has the .inc extension but not .cpp. The Queue.inc should be included at the end of the Queue.h file after the class declaration (this is because we cannot split templated classes into fully separate .h and .cpp files). Also, note that the Queue class must be templated so that it works for various types as applicable and for taking priority or not.

Remember to keep data private, and const to protect objects (explicit and implicit parameters) whenever you can. Your Queue class must contain the same public functions and overloaded operators as the CQueue class, except that parameters for the data stored will now be of some generic type (a template parameter) instead of a char.

Your template specification may look like this:

template< typename DataType, bool useDataAsPriority >

The first template parameter specifies the data type and the second one specifies if the queue considers priority or not. If the templated Queue needs to consider the priority (instead of FIFO), we will use the data itself as the priority and sort the data in an ascending order. Therefore, it is essentially using a sorted list, instead of a simple linked-list. To implement this, you will need to modify the Enqueue function to insert new element at the position that is sorted by the priority and the concatenate operator (operator+) to concatenate two queues based on their priorities. You can read more details about priority queue here.

Below are recommended steps and hints to implement your template class:

  1. Copy your CQueue.h to Queue.h and CQueue.cpp to Queue.inc. Modify it to a template class. (Hint: when you specify a non-template-friend function for your template class, you cannot separate the declaration and definition easily. In this case, you are allowed to put the definition of the friend function directly inside the class definition.)

  2. Modify the Enqueue function to check the second template parameter useDataAsPriority. If useDataAsPriority is true, insert the new node as a sorted list. Otherwise, append it at the tail of the linked-list. (Hint: review in-class exercise 6-1 if needed.)

  3. Modify the operator+ to check the second template parameter useDataAsPriority. If useDataAsPriority is true, concatenate them as a sorted list. Otherwise, concatenate them as usual.

  4. Similar to the previous part, you are given a file named TQueueTest.cpp that tests your implementation (for the templated version). Once you are done implementing Queue.h and Queue.inc, you should compile and run tQueueTest to make sure that all test cases pass. Also keep in mind that these are minimal cases. You should develop your own test cases and test your program thoroughly.

Makefile and Compiling#

Create and submit a Makefile with two rules (aside, perhaps, form the clean rule):

  1. a rule that creates a target executable called cQueueTest for the first part of this homework, and
  2. a rule that creates a target executable called tQueueTest for the second part of this homework (the templated version).

These two targets should compile with no errors or warnings using the typical C++ compilation command: g++ <source> -Wall -Wextra -std=c++11 -pedantic. Do not use any special libraries that require different or additional compiler options.

Git Log and README#

In the assignment's folder of your private repository (not the team repository you made for the midterm project), create a new subfolder named hw7. Do your work in that subfolder and use git add, git commit, and git push regularly. Use git commit and the associated comments to document your work. e.g. if you just modified CQueue.cpp to add the Enqueue functionality, you might do git add CQueue.cpp; git commit -m "Enqueue function is added!"; git push.

Your submission includes a copy of the output of git log showing at least four commits to the repository. Save the git log output into a file called gitlog.txt (e.g. by doing git log > gitlog.txt).

Also, please submit a file called README (not README.txt or README.md, etc - just README) including information about what extra helper functions you added (if any) and anything the graders should know about your submission. In README you should:

  • Write your name and JHED ID at the top of the file
  • Briefly justify the structure of your program; why you defined the functions you added, etc
  • If applicable, highligh anything you did that was particularly clever
  • If applicable, tell the graders if you could not do everything. Where did you stop? What did you get stuck on? What are the parts you already know do not work according to the requirements?

Specific Requirements#

  • Do not eliminate anything from CQueue.h file. You can add more private (helper) functions if you want to, but your CQueue.cpp file should at least provide an implementation for all the functions that are declared in CQueue.h file.
  • Your program should only use new and delete for dynamic memory allocation/deallocation. Do not make use of C functions such as malloc, calloc, free, etc. for that purpose.
  • All variables must be declared inside functions. No variables should be global or extern.
  • Do not use auto.
  • Use header guards in all headers (.h) files.
  • Do not use using in .h or .inc files.
  • In C++ source files (.cpp), you may import individual symbols using statements like using std::string. Do not use using namespace <id>, either in headers or in source files.
  • Make use of gdb to debug and also run valgrind to make sure there is no memory leakage or invalid memory usage regularly. Your submitted program should not have memory leak problem.
  • Do develop your own test cases and test your functions thoroughly.

Create a .zip file named hw7.zip containing:

  • CQueue.h
  • CQueue.cpp
  • Queue.h
  • Queue.inc
  • CQueueTest.cpp
  • TQueueTest.cpp
  • README
  • Makefile
  • gitlog.txt

Copy hw7.zip file to your local machine (using scp or pscp), and submit it as Homework 7 on GradeScope. When you submit, GradeScope conducts a series of automatic tests. These do basic checks, e.g. to check that you submitted the right files. If you see error messages (in red), address them and resubmit. You may re-submit any number of times prior to the deadline; only your latest submission will be graded. Review the course syllabus for late submission policies (grace period and late days).
